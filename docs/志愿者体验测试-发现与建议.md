# 志愿者首次体验测试 —— 发现与改进建议

**日期：** 2026-06-16
**方法：** 用一个**无任何项目背景**的「新人志愿者」角色冷启动浏览本仓库，模拟一位
*会用电脑、有 GitHub 账号、能读中文、但不写代码、只模糊知道「Pull Request」是什么*
的志愿者，从 `README.md` 开始自由点击，尝试完成任务一与任务二的首次贡献，并记录
每一处困惑、犹豫与死胡同。

> 一句话结论：**内容侧（找素材、按模板填）做得相当好；真正会劝退非技术志愿者的，
> 是「最后怎么把东西交上去」这一公里，尤其是提 PR 时的 base/head 方向确认。**

---

## 一、困惑清单（按严重程度排序，最致命在前）

### 1.【最致命】提交 PR 的「最后一公里」：base / head 方向确认看不懂

- **位置：** `CONTRIBUTING.md` → 第三步 → 第 6 步
- **原文：**
  > **关键一步**：确认方向是 **base repository = 原仓库 / base: `main`** ← **head repository = 你的用户名/仓库**。默认通常就是对的。
- **为什么困惑：**
  对一个非技术志愿者来说，`base repository` / `head repository` 是天书般的术语，
  根本不知道哪个是「我的拷贝」、哪个是「原仓库」，更不知道箭头方向代表「把我的改动
  合并回原仓库」。更要命的是这句话**自相抵消**：一边把它标成「关键一步」（=很重要、
  我得盯紧），一边又说「默认通常就是对的」（=你别管）。志愿者会对着 GitHub 那个
  compare 页面的两个下拉框**不敢点**——管也不是，不管也不是。
- **缺什么：**
  ① 没有「如果方向反了/默认错了，我具体该怎么改」的兜底说明；
  ② 整个 Fork→PR 七步**全是纯文字、没有一张截图**，新人无法把文字和真实页面对上，
  任何一步页面长得跟描述不完全一样就会怀疑自己走错。
- **影响：** 这是整个流程里唯一「完全看不懂、又被明确标成关键」的点，是**最可能让人
  在终点线前放弃**的单点故障。

### 2.【内容矛盾】任务一：到底是「必须用真实对话」还是「可以自己写」？

- **位置：** `tasks/01-anchor-book/README.md` 与 `tasks/01-anchor-book/TEMPLATE.md`
- **冲突的两处原文：**
  - 强调真实来源（TEMPLATE「四个要素，缺一不可」之一）：
    > **来源对话** | 来自 `data/dialogs/` 写对话编号……
  - 又允许自己写（README「最缺什么」段）：
    > ⭐ **最缺 1、2、7 分的例子**……所以对 1/2/7 分槽，你可以：……或自己**写**一小段示意性的 1 分 / 7 分摘录（请在理由里注明"示意"）……
- **为什么困惑：**
  通篇反复强调「注明来源、用真实对话」，却在最缺的 1/2/7 分槽突然说「可以自己编」。
  谨慎的新人会反复横跳：**我到底该找真实片段，还是可以自己写？自己写的会不会被嫌弃
  「这不是真实数据」而被退回？** 这种规则上的不确定，会让人干脆不敢下手，或专挑
  最稳妥的 3–6 分槽，反而把「最缺」的极端分继续空着。
- **缺什么：** 一句把政策一次讲清的话——「真实优先；1/2/7 这种现有数据给不出的极端分，
  **明确允许**自己写并标注『示意』，不会被嫌弃」。

### 3.【可选项含糊】`submissions/index.md` 到底要不要我改？

- **位置：** `tasks/02-new-dialogs/README.md` 与 `tasks/02-new-dialogs/submissions/index.md`
- **原文：**
  > 提交新对话时，可顺手在表里加一行（维护者也会维护）。
  >
  > 已提交的对话会记在 `submissions/index.md`。
- **为什么困惑：**
  「**可**顺手加」「维护者**也会**维护」——「可」和「也会」让人判断不了这到底是
  **我的义务还是可选**。对新人而言，「顺手加一行表格」=又要改第二个文件、又要懂
  Markdown 表格语法，听起来就累。结果是：**不确定不改会不会被退回**，于是要么硬着
  头皮多做一步，要么忐忑地跳过。
- **缺什么：** 明确写「**这一步可选，不改也完全没关系，维护者会补**」，卸下心理负担。

### 4.【操作地雷】任务二文件名：能用中文，但不能带斜杠

- **位置：** `CONTRIBUTING.md`（任务二新建文件那段）
- **原文：**
  > 在文件名框里输入文件名（如 `good-情绪命名.md`，文件名用中文没问题，**不要**带路径或斜杠）。
- **为什么困惑：**
  指令本身能照做，但「**不要**带斜杠」这种「别犯错」式提醒，恰恰说明这是个**真容易
  踩的坑**——GitHub 网页「新建文件」那个文件名框对新手并不直观（输入 `a/b.md` 会
  悄悄创建子文件夹）。新人心里没底：到时候一紧张会不会就打错、创建出一个怪文件夹。
- **缺什么：** 一句正面示范 + 后果说明，例如「直接在框里打 `good-xxx.md` 就行；
  如果手滑输入了带 `/` 的名字，GitHub 会建出一个子文件夹，删掉重来即可」。

### 5.【门槛感】README 开头的统计术语

- **位置：** `README.md` 开头「为什么需要你」
- **原文（涉及术语）：** 「评分者差异占了总方差的 **54%**」「ICC ≈ 0」「截尾均值」
- **为什么困惑：**
  ICC、占总方差 54% 这些词，会让一部分人一开始就觉得「这是给专业人士的项目，
  不是给我的」。**好在每条都配了人话翻译**（「换个人打分结果就变」），所以并不致命，
  排在最后。
- **缺什么：** 可选——把术语进一步弱化或挪到脚注/链接，正文只留大白话。

## 二、做得好、不建议改动的地方

- **「修一把测不准的尺子」的比喻**贯穿全文，新人 30 秒就懂项目在干嘛、自己能帮什么。
- **反复强调「不需要写代码、不需要命令行」**，精准打掉了非技术志愿者最大的恐惧。
- **任务一 TEMPLATE 给了一个能直接照抄、且能在 `data/dialogs/dialog-10.md` 里核对到
  的真实例子**，大幅降低「我该写成什么样」的门槛。
- **任务二 `submissions/EXAMPLE.md` 是一段真正像样的「冷漠说教」坏对话**，一看就懂
  什么叫「明显差」；`data/low-rating-pool/low-001.md` 等现成可用。
- **PR 模板（`.github/PULL_REQUEST_TEMPLATE.md`）会自动弹出、勾选式、且提醒脱敏**，
  最后一关其实是软着陆（只是志愿者提前找它会费点劲，因为藏在 `.github/`）。
- **仓库内相对链接全部可达**，点哪都能到，没有死链。
- **反复提醒「两套评分别混淆」「请脱敏」「尊重对话当事人」**，边界说得很清楚。
